# ADR-0007: Maintain a Monad Lock or State File

## 1. Status

approved, accepted.

This ADR records a decision already established by the approved Monad product charter: Monad shall maintain a lock or state file.

## 2. Context

Monad is a Rust-native polyglot repo runtime and developer-experience CLI.

Monad uses `monad.toml` as its canonical repository-level manifest for user-authored or user-reviewed intent.

However, `monad.toml` should not become a dumping ground for every resolved, generated, observed, or machine-derived detail.

Monad will eventually need to track information such as:

1. Resolved manifest schema version.
2. Effective workspace configuration.
3. Enabled adapters.
4. Adapter-detected projects.
5. Native toolchain versions.
6. Native manifest fingerprints.
7. Project graph fingerprints.
8. Task graph fingerprints.
9. Generated file provenance.
10. Last known verification state.
11. Repo evolution operation metadata.
12. Template versions.
13. Policy pack versions.
14. AI context generation metadata.
15. AI memory index metadata.
16. Release workflow metadata.
17. Migration state.
18. Compatibility information.

Some of this information should be versioned and reproducible. Some of it may be local, ephemeral, or machine-specific.

Monad needs a formal state model so that user-authored intent, generated reproducibility metadata, and local cache data do not become confused.

## 3. Decision

Monad shall maintain a generated lock or state artifact separate from `monad.toml`.

The conceptual distinction shall be:

```text
monad.toml        -> user-authored or user-reviewed intent
monad.lock        -> generated, reviewable, reproducibility-oriented resolved state
.monad/           -> optional local/generated operational state, cache, and reports
```

The initial canonical lock file shall be:

```text
monad.lock
```

The initial lock file should live at the repository root.

The initial local state directory, when needed, shall be:

```text
.monad/
```

The exact contents of `.monad/` may be introduced gradually and should not be required for the earliest CLI scaffold.

The initial policy shall be:

1. `monad.toml` is hand-authored or user-reviewed intent.
2. `monad.lock` is generated by Monad and should be reviewable.
3. `monad.lock` should be committed for repositories that want reproducible Monad behavior.
4. `.monad/` may contain local state, cache files, reports, generated indexes, or machine-specific runtime artifacts.
5. Local ephemeral `.monad/` contents may be ignored by Git unless explicitly designated as committed project state.
6. Monad must not rely on hidden state when a visible, reviewable lock artifact is required for reproducibility.
7. Lock/state format details should be versioned.
8. Monad commands should explain when the lock/state artifact is missing, stale, invalid, or regenerated.

## 4. Rationale

Monad needs a generated state artifact because it is more than a static manifest parser.

Monad is intended to inspect repositories, coordinate native tools, generate files, build graphs, record provenance, enforce policy, support AI-readable context, and evolve repository structure safely.

Many of those operations require resolved or observed data.

Putting all of that information into `monad.toml` would be wrong because `monad.toml` should remain a clean statement of intent.

A generated lock or state file gives Monad a place to record resolved state without polluting user-authored configuration.

A root-level `monad.lock` is appropriate because it is visible, reviewable, and consistent with the idea that reproducibility-critical state should not be hidden.

A `.monad/` directory is appropriate for local or generated operational state that may be too large, too volatile, or too machine-specific for the root lock file.

This split gives Monad a strong foundation:

```text
intent      -> monad.toml
resolution  -> monad.lock
operations  -> .monad/
```

## 5. Alternatives Considered

### 5.1 Store Everything in `monad.toml`

Monad could store both intent and resolved state inside `monad.toml`.

This was rejected.

Advantages:

1. Fewer files.
2. Simpler first implementation.
3. One obvious place to inspect configuration.

Disadvantages:

1. Mixes user intent with generated state.
2. Creates noisy diffs.
3. Makes the manifest harder to understand.
4. Risks overwriting user-authored comments and formatting.
5. Makes generated provenance harder to distinguish from deliberate configuration.
6. Makes schema evolution more dangerous.
7. Violates the manifest-as-intent principle.

### 5.2 Use Only Hidden State Under `.monad/`

Monad could store all generated state under:

```text
.monad/
```

This was rejected as the only state mechanism.

Advantages:

1. Keeps the root cleaner.
2. Allows multiple state files.
3. Avoids a root lockfile.
4. Works well for caches and generated reports.

Disadvantages:

1. Less visible.
2. Easier to accidentally ignore all important state.
3. Worse for reproducibility-critical data.
4. Worse for reviewability.
5. Less friendly to humans and AI assistants scanning the repo.
6. Can blur the line between cache and durable project state.

`.monad/` remains useful, but it should not be the only place for reproducibility-oriented state.

### 5.3 Use `monad-state.toml`

Monad could use a root file named:

```text
monad-state.toml
```

This was considered but not selected as the primary name.

Advantages:

1. Explicitly describes state.
2. Avoids the term lock if the file is not strictly deterministic.
3. Human-readable.

Disadvantages:

1. Less conventional than a lockfile.
2. May imply manually edited state.
3. Less clearly tied to reproducibility.
4. More awkward as a generated artifact name.

The term `lock` better communicates that the file is generated and reproducibility-oriented.

### 5.4 Use `monad.lock.toml`

Monad could use:

```text
monad.lock.toml
```

This was considered.

Advantages:

1. Makes the TOML format explicit.
2. Editor tooling may recognize it more readily.
3. Clearer for humans.

Disadvantages:

1. More verbose.
2. Less conventional as a lockfile name.
3. Locks the visible name to a specific serialization format.
4. Reduces flexibility if the format changes later.

The selected name `monad.lock` leaves room for the internal format to evolve while still allowing the first version to be TOML, JSON, or another structured format if implementation requires.

### 5.5 Use JSON Only

Monad could use a JSON lock file.

This was deferred.

Advantages:

1. Machine-readable.
2. Easy to parse.
3. Good for generated state.
4. Stable for tooling.

Disadvantages:

1. Poorer human review experience.
2. No comments.
3. Less aligned with the TOML-based manifest.
4. Less pleasant for governance-grade review.

The exact lock file serialization should be decided during schema implementation.

### 5.6 Use No Lock or State File

Monad could recompute everything on each run.

This was rejected.

Advantages:

1. No generated state.
2. No lockfile maintenance.
3. No stale state problems.
4. Simpler mental model at first.

Disadvantages:

1. Less reproducibility.
2. No durable provenance.
3. Harder to detect drift.
4. Harder to support safe repo evolution.
5. Harder to support AI context generation.
6. Harder to compare current state to prior state.
7. Harder to support future graph or policy workflows.
8. More expensive repeated inspection.

Monad needs a durable state model.

## 6. Consequences

### 6.1 Positive Consequences

Maintaining a lock or state file gives Monad:

1. Clear separation between intent and resolved state.
2. Better reproducibility.
3. Better provenance.
4. Better drift detection.
5. Better reviewability.
6. Better AI-readable project memory.
7. Better support for repo evolution.
8. Better support for graph and policy workflows.
9. Better support for migration.
10. Better foundation for CI/local parity.
11. Better support for diagnostics.
12. Better support for future release workflows.

### 6.2 Negative Consequences

This decision also creates costs:

1. Monad must design a lock/state schema.
2. Monad must handle stale lock files.
3. Monad must decide when to regenerate state.
4. Monad must avoid noisy diffs.
5. Monad must document what should be committed.
6. Monad must distinguish durable state from local cache.
7. Monad must provide good diagnostics for invalid state files.
8. Monad must avoid hiding important behavior in generated files.
9. Monad must avoid letting the lock file become too large.
10. Monad must avoid using local machine-specific data in committed lock state.

### 6.3 Neutral Consequences

This decision does not fully define the lock file schema.

This decision does not fully define the `.monad/` directory layout.

This decision does not decide whether `monad.lock` uses TOML, JSON, or another internal format.

This decision does not require every command to write state immediately.

This decision does not require `.monad/` to exist in the first implementation slice.

This decision does not replace native ecosystem lockfiles such as `Cargo.lock`, `bun.lock`, `package-lock.json`, `pnpm-lock.yaml`, `poetry.lock`, or similar files.

Native ecosystem lockfiles remain authoritative for their ecosystems.

## 7. Lock and State Authority

The authority model shall be:

```text
monad.toml       -> explicit Monad-level intent
monad.lock       -> generated resolved Monad-level state
.monad/          -> local/generated operational state, cache, reports, and indexes
native manifests -> native ecosystem intent
native lockfiles -> native ecosystem dependency resolution
```

Examples:

1. `Cargo.toml` defines Rust package/workspace intent.
2. `Cargo.lock` records Rust dependency resolution.
3. `package.json` defines JS/TS package intent.
4. `bun.lock` records Bun dependency resolution.
5. `monad.toml` defines Monad-level repo intent.
6. `monad.lock` records Monad-level resolved state.

Monad must not attempt to replace native ecosystem lockfiles.

## 8. Initial `monad.lock` Responsibilities

The initial `monad.lock` should eventually record enough information to support reproducibility and drift detection.

Potential contents include:

1. Lock schema version.
2. Monad CLI version.
3. Source manifest fingerprint.
4. Effective configuration fingerprint.
5. Adapter list.
6. Adapter versions or capability versions.
7. Discovered project identifiers.
8. Native manifest fingerprints.
9. Native lockfile fingerprints.
10. Generated artifact fingerprints.
11. Template pack identifiers.
12. Policy pack identifiers.
13. Graph fingerprint.
14. Last successful sync metadata.
15. Last generated AI context metadata.
16. Provenance references.
17. Migration metadata.

The first implemented lock file may be much smaller.

The lock schema should start minimal and grow only when implementation needs justify new fields.

## 9. Initial `.monad/` Responsibilities

The `.monad/` directory may eventually contain generated operational state such as:

```text
.monad/
  cache/
  reports/
  state/
  provenance/
  context/
  graph/
  tmp/
```

This is illustrative only.

Possible responsibilities include:

1. Local caches.
2. Generated reports.
3. Diagnostic reports.
4. Graph exports.
5. AI context packs.
6. Provenance logs.
7. Migration reports.
8. Temporary files.
9. Machine-specific observations.
10. Tool execution traces.

The `.monad/` directory should not become a hidden source of required user intent.

If a file is required for reproducible project behavior, it should either live in a visible committed location or be clearly documented as committed state.

## 10. Git Tracking Policy

The recommended initial Git policy is:

1. `monad.toml` should be committed.
2. `monad.lock` should usually be committed for application and product repositories.
3. `.monad/cache/` should be ignored.
4. `.monad/tmp/` should be ignored.
5. `.monad/reports/` may be ignored unless reports are explicitly committed.
6. `.monad/context/` may be committed if used as repo-native AI context.
7. `.monad/provenance/` may be committed if it represents durable project history.
8. The exact `.gitignore` policy should be documented when `.monad/` is introduced.

Monad should eventually generate or recommend `.gitignore` entries that distinguish committed state from local cache.

## 11. Staleness Policy

Monad must be able to detect stale lock or state artifacts.

A lock file may be stale when:

1. `monad.toml` changes.
2. Native manifests change.
3. Native lockfiles change.
4. Adapter configuration changes.
5. Template versions change.
6. Policy settings change.
7. Project structure changes.
8. Monad CLI version changes in a schema-relevant way.
9. Generated artifacts are modified externally.
10. Schema version changes.

When state is stale, Monad should report it clearly.

Possible behaviors:

1. Warn and continue.
2. Fail for strict commands.
3. Regenerate with explicit command.
4. Regenerate automatically only when safe.
5. Require `monad sync` to reconcile state.

The exact behavior should be command-specific and may be refined later.

## 12. Relationship to `monad sync`

`monad sync` should eventually be the primary command for reconciling manifest intent with generated state and repository files.

Possible responsibilities:

1. Validate `monad.toml`.
2. Compute effective configuration.
3. Refresh `monad.lock`.
4. Update generated files.
5. Update AI context files.
6. Update graph outputs.
7. Update provenance records.
8. Detect drift.
9. Present dry-run previews.
10. Apply safe changes.

`monad sync --dry-run` should show intended state and file changes before writing where practical.

## 13. Relationship to `monad doctor`

`monad doctor` should inspect lock/state health.

It should eventually report:

1. Whether `monad.toml` exists.
2. Whether `monad.lock` exists.
3. Whether `monad.lock` is stale.
4. Whether lock schema is supported.
5. Whether native toolchains match expected state.
6. Whether native manifests changed.
7. Whether generated files drifted.
8. Whether local cache is corrupt.
9. Whether AI context is stale.
10. Whether provenance references are valid.

## 14. Relationship to `monad check`

`monad check` should use lock/state data carefully.

It may use lock/state artifacts to:

1. Validate repo consistency.
2. Detect drift.
3. Select projects.
4. Compare expected and observed state.
5. Verify generated artifacts.
6. Verify policy compliance.
7. Confirm AI context freshness.
8. Confirm graph outputs are current.

However, `monad check` should not blindly trust stale generated state.

It should validate freshness before relying on it for consequential checks.

## 15. Relationship to AI Context and Memory

Monad’s AI-readable context and full AI memory system will benefit from lock/state artifacts.

The state model may support:

1. Current-state files.
2. AI bootstrap prompts.
3. Repo summaries.
4. Handoff packets.
5. Graph summaries.
6. Command inventories.
7. Verification reports.
8. Memory indexes.
9. Provenance references.
10. Drift reports.

AI context should be generated from real repository inspection, accepted docs, ADRs, manifest intent, and verified state rather than guessed chat memory.

The lock/state model should help make AI context reproducible and inspectable.

## 16. Relationship to Provenance

Monad should record provenance for generated or modified artifacts.

Provenance may include:

1. Command name.
2. Command version.
3. Timestamp.
4. Input manifest fingerprint.
5. Template identifier.
6. Generated paths.
7. Changed paths.
8. Dry-run or applied mode.
9. User approval mode.
10. Verification result.
11. Related work packet.
12. Related ADR or requirement references.

Some provenance may belong in `monad.lock`.

More detailed provenance may belong under `.monad/provenance/` or another approved path.

The exact provenance model should be defined later.

## 17. Implementation Guidance

### 17.1 Initial Lock File Name

The root lock file name shall be:

```text
monad.lock
```

### 17.2 Initial State Directory Name

The optional local/generated state directory shall be:

```text
.monad/
```

### 17.3 Intent Versus State

Implementation must preserve this rule:

```text
Never treat generated state as user-authored intent.
```

If generated state conflicts with `monad.toml`, Monad should report the conflict and provide a safe reconciliation path.

### 17.4 Safe Writes

Writing `monad.lock` or `.monad/` state should follow Monad’s safe file operation principles:

1. Avoid destructive writes.
2. Use atomic writes where practical.
3. Preserve user files.
4. Avoid unnecessary diffs.
5. Support dry-run for commands that would update state.
6. Explain what changed.
7. Record provenance where appropriate.

### 17.5 Minimal First Implementation

The first implementation does not need a complete lock schema.

A minimal first lock model may contain:

1. `schema_version`
2. `monad_version`
3. `manifest_fingerprint`
4. `generated_at`
5. `workspace_root`
6. `adapters`

Even this may be deferred until manifest loading exists.

This ADR authorizes the state model. It does not require fully implementing it in the first scaffold commit.

### 17.6 Serialization Format

The serialization format for `monad.lock` should be selected during implementation.

Acceptable initial candidates:

1. TOML
2. JSON

TOML is more human-readable and aligned with `monad.toml`.

JSON is more machine-oriented and common for generated state.

The implementation should choose the format that best balances readability, stability, and machine parsing.

If the choice becomes consequential, it should be recorded in a later ADR or schema specification.

### 17.7 Fingerprinting

Monad should use stable fingerprints for drift detection.

Fingerprints may eventually be computed from:

1. `monad.toml`
2. native manifests
3. native lockfiles
4. generated files
5. templates
6. policy packs
7. adapter capability metadata
8. graph inputs

Fingerprinting details should be deterministic and documented.

### 17.8 Avoid Machine-Specific Committed State

Committed lock state should avoid machine-specific absolute paths, usernames, hostnames, timestamps that churn unnecessarily, or local-only tool paths.

Machine-specific observations should be local state, not reproducibility lock state.

### 17.9 Schema Versioning

Both `monad.lock` and `.monad/` state files should include schema versions where appropriate.

Monad should produce clear diagnostics for:

1. Missing lock file.
2. Invalid lock file.
3. Unsupported lock schema.
4. Stale lock file.
5. Conflicting state.
6. Corrupt local state.
7. State generated by unsupported Monad versions.

## 18. Verification

This ADR is satisfied when Monad can eventually:

1. Distinguish `monad.toml` intent from generated state.
2. Recognize whether `monad.lock` exists.
3. Generate or refresh `monad.lock` when appropriate.
4. Detect stale lock state.
5. Report missing lock state clearly.
6. Report invalid lock state clearly.
7. Avoid requiring `.monad/` for basic CLI startup.
8. Use `.monad/` only for documented generated/local state.
9. Avoid storing machine-specific committed state in `monad.lock`.
10. Include tests for missing, valid, invalid, and stale lock/state scenarios.

Initial future verification commands may include:

```bash
cargo test --workspace lock
cargo test --workspace state
cargo test --workspace
cargo clippy --workspace --all-targets --all-features -- -D warnings
```

Once the root verification script exists, the preferred command should be:

```bash
./tools/scripts/verify.sh
```
